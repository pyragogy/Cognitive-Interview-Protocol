# Contributing

> The Cognitive Interview Protocol is an open research project.
> Contributions are welcome — but what "contribution" means here is different from what it means in a software project.

---

## What kinds of contribution are useful at this stage

The protocol is at v0.2. It has not been tested in real interview sessions. At this stage, the most valuable contributions are analytical, not implementational.

**Critique of the protocol design.** If the interview structure, output format, or review process has a structural flaw, we want to know. A well-reasoned critique — one that identifies a specific problem and explains why it matters — is more useful than a proposed fix.

**Alternative framings of the research questions.** The questions in `README.md` and `docs/ROADMAP.md` reflect one way of thinking about the problem. If you see a different framing that changes what we should be investigating, that is worth discussing.

**Annotated interview transcripts.** If you conduct an interview session using the protocol (or a close approximation of it) and are willing to share the transcript, it is useful as evidence — even if the session did not go well. A transcript where the protocol broke down is particularly useful.

**Counterexamples.** Cases where the approach would not work, or where the underlying assumptions do not hold. These are research contributions.

**Connections to existing literature.** Many of the `[VERIFY SOURCE]` markers in the documentation represent gaps in our literature review. If you have a reference that fills one of those gaps, that is a useful contribution.

---

## What to do with a contribution

1. Open an issue describing the contribution and its context.
2. If the contribution is a document (critique, analysis, transcript), attach it or link to it in the issue.
3. If the contribution modifies an existing document in the repository, open a pull request referencing the issue.

There is no formal review timeline. This is a research project, not a software project with release schedules.

---

## What is not useful at this stage

- Code contributions, unless they implement a specific piece of the protocol that is explicitly marked as ready for implementation.
- Proposals for alternative technologies (different knowledge graph formats, different AI models). The protocol is model-agnostic by design; technology questions are secondary.
- Feature requests for a software system that does not yet exist.

---

## Authorship and attribution

Contributions that substantively shape the protocol will be acknowledged in the repository and in any publications that draw on the work. The form of attribution will depend on the nature and extent of the contribution.

If you are not sure whether your contribution warrants attribution, ask in the issue.

---

## Code of conduct

This project expects contributors to engage seriously and in good faith with the research questions. Critique is welcome; dismissiveness is not. If you think the project is asking the wrong question, explain why — that is a useful contribution. If you want to dismiss the project without engaging with it, there are better uses of your time.

---
